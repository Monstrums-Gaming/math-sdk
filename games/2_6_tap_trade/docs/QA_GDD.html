Debug handlesDEV BUILDS ONLY: window.__tt (.simTime(), .cfg, .tapState.chips) and window.__ttHistory (entries). Their absence in a production build is itself a test (TT-ERR-07).
Reset statelocalStorage.clear() removes taptrade.* keys (betSize, theme, muted, seenHint, speed, tapConfirm). Balance/history reset on reload (in-memory).

Pointer profile matters: tap-to-confirm defaults ON for coarse pointers (touch) and OFF for mouse. When a single click doesn’t place a chip, check Settings → “Tap to confirm” before filing — the first tap only previews (TT-SET-05).

4Wallet & Ledger (TT-LED) — the critical suite

The displayed balance is an optimistic ledger. The only legal movements:

place chip      →  balance − stake        (immediately, exactly once)
win REVEAL      →  balance + payout       (at the line-fill moment, not at settlement)
loss reveal     →  no change              (the stake already left at placement)
quiescence      →  balance := server RGS balance   (silent reconcile; normally a no-op)
IDScenarioExpected
TT-LED-01Place one chip, let it LOSEOne deduction at placement; NO second deduction at the explode animation; “Missed” float is a marker, not a debit
TT-LED-02Place one chip, let it WINCredit lands exactly when the line fills the cell — throttle the network (LIVE/mock) to prove the gap between settlement and reveal shows no early credit
TT-LED-033 chips across 2 columns, mixed outcomes3 immediate deductions; credits only at each reveal; after the last reveal the pill EQUALS the server balance to the cent (LIVE: last /wallet/* response balance.amount ÷ 1e6)
TT-LED-04Out-of-order settlement (chips in far column settle while near column reveals)The pill NEVER jumps to a stale per-round snapshot — regression for the “$1,010 vs $1,005” drift class
TT-LED-05Force /wallet/play failure (devtools block) after placingChip removed from the board, stake refunded on the pill, error modal/toast shown; LATER chips unaffected
TT-LED-06Session strip mathP/L = Σ(payouts) − Σ(stakes); “$X in play · N chips” counts only unresolved chips; pips cap at 8, newest first with a ring
TT-LED-07Mixed bet sizes queued (e.g. $3 then $7 chips)Each /wallet/play body carries ITS chip’s amount (×1,000,000) and mode — verify in the network tab

File as a launch blocker: any double deduction, any early win credit (before the line reaches the cell), or any post-session mismatch between the pill and the RGS balance.

11Math Spot-Checks & Reference

11.1 Network-level checks (mock or LIVE)

  • Every /wallet/play body: mode = "call_<cents>" matching the tapped cell’s displayed multiplier ×100; amount = stake × 1,000,000.
  • Win round: round.active:true in the play response, then exactly one /wallet/end-round. Loss round: round.active:false, NO end-round call.
  • Book events in round.state: win = cellCall → wincap → finalWin (wincap on EVERY win — see Not-A-Bug #1); loss = cellCall → finalWin(amount:0).
  • Scales: event amounts are bet-cents (450 = 4.5×); round.payoutMultiplier is PLAIN (4.5); API money ×1,000,000.

12Not-A-Bug List (do not file)

#ObservationWhy it’s intentional
1A wincap event on every winning round, even a 1.4× winPer-mode max_win equals the mode’s own multiplier (math-side design); the client no-ops it
2Only 2 chips per column, even with a gap between cellsStatic cap — a 3rd chip can demand an undrawable win-lose-win pattern; a dynamic cap would leak drawn outcomes
3Balance doesn’t change when a chip explodesThe stake left at placement; “Missed” is an outcome marker, not a debit
4Win credit waits for the line to reach the cellReveal-time crediting is the masking design — an early credit would spoil the outcome
5The same multiplier appears in different places over timeCell pricing follows a live volatility model; only the VALUES are fixed (the ladder)
6Game feels faster than real time at speed 1×Base sim pace is 1.3× by design; the setting multiplies it
7History clears on reloadLanding-shot JPEGs are too large for localStorage — in-memory only
8Balance pill isn’t clickable (demo resets on click in LOCAL)The app’s wallet is mock/RGS-owned; no reset affordance on purpose
9No “Try again” button on errors (demo has one)The app compiles its odds in — there is no retryable load; Reload covers recovery
10Resume shows no chip on the boardBooks are position-neutral; the tapped cell is never persisted
11Prices/line movement don’t match any real marketThe chart is presentation over pre-generated outcomes (certified replay); prices are theatre
